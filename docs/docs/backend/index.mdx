# Backend: index.js

The backend is a Node.js process that owns the network. The UI never touches the network directly — it sends commands to the backend and receives events back.

## Pipe setup

```javascript reference
https://github.com/Ginowine/p2p-pong/blob/master/index.js#L6-L12
```

`Bridge` mounts the `ui/` folder inside the app window. `runtime.start()` returns a **duplex pipe** — writing to it sends data to the UI, and data the UI writes comes back here. `Pear.teardown` cleans up when the app closes.

## Message framing

```javascript reference
https://github.com/Ginowine/p2p-pong/blob/master/index.js#L14-L28
```

Because TCP (and the pipe) is a **stream**, messages can arrive split across multiple chunks. The backend accumulates incoming bytes in `recvBuf`, splits on newlines, and only processes complete lines. Each message is a JSON object followed by `\n`.

## Message router

```javascript reference
https://github.com/Ginowine/p2p-pong/blob/master/index.js#L37-L46
```

Every message from the UI has a `type`. This switch dispatches to the appropriate handler. The separation keeps each responsibility isolated and easy to extend.

## Hosting a game

```javascript reference
https://github.com/Ginowine/p2p-pong/blob/master/index.js#L48-L62
```

`topic` is a random 32-byte value that acts as a **rendezvous point** on the DHT. Any peer who knows this topic can find and connect to the host. `flushed()` waits until the DHT has been told about us. The topic is sent back to the UI as a 64-character hex string — that's the room code the player shares with their opponent.

## Joining a game

```javascript reference
https://github.com/Ginowine/p2p-pong/blob/master/index.js#L64-L71
```

The joiner converts the room code back to bytes and asks the DHT to find a peer advertising that topic. No `flushed()` wait is needed here — the connection arrives via the `'connection'` event.

## Peer connection and framing

```javascript reference
https://github.com/Ginowine/p2p-pong/blob/master/index.js#L92-L117
```

Peer messages use **length-prefixed framing**: a 4-byte big-endian integer giving the payload size, followed by the payload bytes. This is more robust than newline-delimited framing for binary-safe data over a raw TCP stream.

```javascript reference
https://github.com/Ginowine/p2p-pong/blob/master/index.js#L73-L84
```

## Leaderboard (Hypercore)

```javascript reference
https://github.com/Ginowine/p2p-pong/blob/master/index.js#L130-L170
```

Hypercore stores entries as an **append-only log** on disk. Each entry gets a permanent index. `lbCore.discoveryKey` is a unique identifier derived from the core's keypair — peers who join this swarm topic will replicate the same log, keeping leaderboards in sync across machines. `live: true` means replication stays open and updates stream automatically.
